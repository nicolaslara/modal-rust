The local-build coupling problem

`modal-rust run` obtains arch-independent information (the entrypoint manifest) by an arch-dependent means (compiling and executing the user's crate locally). This page maps every way that coupling can hurt, and every way out. Written 2026-06-11, from the dict/queue live-debugging session.

1. The system as it stands — who runs cargo, where

Laptop (any OS/arch)

  • Describe build: cargo build -p <pkg> --bin modal_runner, DEBUG, shares the dev target/; output exec'd as runner --describe → manifest (entrypoints + decorator config).
  • Skipped on a describe-cache hit (source + Cargo.lock hash).
  • Input validation: runner --check-input, degrades to the remote check if unavailable.
  • Nothing built here is ever uploaded.

Modal container — RUN (linux)

  • Wrapper checks the runner-binary cache (volume, keyed on source hash + package): hit ⇒ no cargo at all.
  • Miss ⇒ unpack cache archive (registry + RELEASE target/), cargo build --release incrementally, store the new binary, re-pack only if something compiled.
  • Cache archive is container-built; never leaves Modal.

Modal image builder — DEPLOY

  • RUN cargo build --release at image-build time; binary baked to /app/modal_runner.
  • Deployed runtime never runs cargo.

What crosses the wire: pruned source only (~1.4 MB for this workspace, sha-deduped, target/ ignore-pruned). Local DEBUG artifacts and the container's RELEASE cache are disjoint worlds; the only thing they share is resolving the same Cargo.lock.

Why the local build exists at all. The CLI cannot link the user's crate, and the manifest lives in the compiled binary (the #[function] macro registers entrypoints + config via inventory, collected at runtime by --describe). Compiling locally is the cheapest general way to evaluate that registration exactly as rustc sees it — including cfgs, feature resolution, and macro expansion. Every alternative below trades away some of that fidelity or pays for it elsewhere.

2. Failure modes

Severity classes: BLOCKS the run with an error; SILENT — produces a wrong manifest without an error (worse); COST — correctness fine, time/resources wasted.

F1 — Compile-time platform dependencies BLOCKS

Likelihood: the headline case. Affects: linux-only -sys crates, bindgen against system headers, proprietary SDKs, crates with compile_error! on non-linux.

The crate compiles for the container (linux) but not on the laptop (macOS/Windows). The local describe build fails with a cargo error before any Modal interaction, even though the remote build would succeed. Mitigating reality: Rust GPU crates mostly follow the cudarc pattern — compile everywhere, link/JIT CUDA at runtime — so burn-add and cuda-vector-add build on a CUDA-less Mac. The blocked class is compile-time coupling, not GPU per se.

F2 — cfg-divergent registries SILENT

Likelihood: low but insidious. Affects: #[cfg(target_os = "linux")]-gated #[function]s (or feature-gated ones with locally-different feature resolution).

The crate compiles on both sides but registers different entrypoint sets. A linux-only #[function] is absent from the locally-built manifest → the CLI reports "unknown entrypoint" for a function that exists remotely. Inverse: a mac-visible entrypoint validates locally, then the container's runner rejects it. No error points at the real cause. Today's partial guards: input validation already degrades on unknown_entrypoint (treats the local runner as non-authoritative), but manifest lookup does not.

F3 — Host-environment-dependent build scripts & macro inputs SILENT

Likelihood: rare. Affects: build.rs probing the host (paths, installed tools, env! at compile time) feeding entrypoint names or config.

Manifest content baked from host state differs from what the container build would produce. Same class as F2, reached through build scripts instead of cfg.

F4 — Local toolchain mismatch BLOCKS

Likelihood: common for newcomers. Affects: missing/old rustc, missing targets/components, rustup not installed.

The container installs its own pinned toolchain; the laptop might not satisfy the crate's MSRV or rust-toolchain.toml. Local failure, remote fine. Distinct from F1 because the fix ("install/update rust") is reasonable to demand — doctor --rust already diagnoses it — but it is still the same structural coupling.

F5 — Env-dependent describe output SILENT

Likelihood: low. Affects: config influenced by env vars at describe time.

The manifest is read on the laptop under the laptop's env. Decorator config is static today (macro literals), so this is currently theoretical — but any future "config from env/file at registration time" feature would make local-vs-container env divergence a silent manifest skew. Worth a standing design rule: registration must stay a pure function of source + features.

F6 — First-build cost on weak machines COST

Likelihood: every new user, once.

The first describe build compiles the user's crate + the light facade (~9 crates) in DEBUG. Small for function-only crates (the light feature split exists for exactly this), but a heavy user crate pays its own compile locally even when only running remotely. Repeats only on source change (describe cache).

F7 — Broken sibling code in the workspace BLOCKS (partially solved)

Likelihood: medium in monorepos.

-p <pkg> scoping plus the cargo-metadata closure means an unrelated broken sibling does not block — only the target package's dependency closure must compile. A broken crate inside the closure blocks both sides equally (not a coupling failure; the remote build would fail too). Listed for completeness because users perceive it as "the CLI is building too much".

3. Alternative solutions

S1 — Manifest override: --no-local-build + flags or --manifest file.json recommended first

Effort: small (CLI-only). Covers: F1, F4, F6 (opt-in), F7.

Skip the describe build entirely; take the entrypoint name from the command and the config from inline flags (--gpu T4 --timeout 600 …) or a user-written manifest file (the modal-rust/describe@1 schema already exists and is documented). Input validation uses its existing degrade path (remote decode check).

ProsCons
Tiny, reuses existing schema + degrade paths; unblocks any crate immediately; also the natural CI/scripting interface.Manual: user re-states config that the decorator already declares (drift risk between decorator and flags); doesn't help F2/F3 (the silent class) — the user's hand-written manifest is just another locally-produced claim.

S2 — Remote describe bootstrap recommended second

Effort: medium. Covers: F1, F2, F3, F4, F5, F6 — the whole class, including SILENT.

When the local build fails (or on --remote-describe): provision a default-config function once, run --describe inside the container, cache the returned manifest locally keyed on the source hash. The chicken-and-egg (config is needed to create the function) is broken by using defaults for the bootstrap function only — --describe needs no GPU, no special timeout. Subsequent runs hit the cached manifest like any describe-cache hit.

ProsCons
The container's own answer is ground truth — kills the silent failures (F2/F3/F5) that nothing else catches; fully automatic; piggybacks on the runner-binary cache (the bootstrap build is reused by the real run).First run pays a remote build before config discovery (~minutes cold); costs money; needs care that the bootstrap function's default resources can build the crate (big crates may need more memory — bootstrap should use a generous CPU/memory default).

S3 — Static source scanning (no compile) rejected

Effort: medium-high, permanent maintenance. Covers: F1, F4, F6 — but introduces its own silent class.

Parse the source with syn for #[modal_rust::function] attributes and extract names + config without compiling. Why rejected: it re-implements rustc's view of the crate — cfg evaluation, feature resolution, include!, macro-generated functions, renamed imports of the macro — and every divergence is a new silent failure mode of exactly the kind S2 eliminates. A scanner that is "usually right" is worse than a build that is honestly wrong.

S4 — Local linux container for describe (docker) parked

Effort: medium. Covers: F1, F2, F3, F4 — when docker exists.

Run the describe build inside a local linux container matching the remote target. Faithful (same OS) and offline-capable, but adds a docker dependency to a tool whose pitch is "no extra runtime deps", costs a cold container toolchain, and macOS docker is qemu/VM-slow for compiles. Strictly dominated by S2 for correctness and by S1 for convenience; worth revisiting only if remote-describe cost proves painful.

S5 — Cross-compiling locally for the container target rejected for this problem

Effort: high (zig/cross toolchains).

Solves a different problem (shipping a locally-built binary, skipping the remote build). For manifest discovery it inherits F1 in reverse (now the laptop must cross-compile linux-only deps) and adds toolchain burden. Tracked separately as the "precompile/builder" exploration — see precompile-builder-review.html.

S6 — Authoring guidance: the cudarc pattern documentation duty

Effort: docs only. Prevents: most real-world F1.

Recommend compile-everywhere/link-at-runtime crates (cudarc, NVRTC JIT) and #[cfg]-free entrypoint registration in the getting-started guide. Best dev loop regardless of which escape hatch exists, because it keeps .local() and the describe build working everywhere.

S7 — Manifest verification mode cheap adjunct to S2

Effort: small once S2 exists. Covers: F2/F3/F5 detection (vs. S2's correction).

modal-rust run --verify-manifest (or a live-tier test): compare the locally-derived manifest against the container's --describe answer and fail loudly on divergence. Turns the silent class into a diagnosable error for users who keep the fast local path.

5. Recommendation

Policy ladder, in order:

  1. Default: local describe build (fast, faithful for the overwhelmingly common compile-everywhere crate), with the describe cache as today.
  2. On local build failure: print a diagnosis that names this exact coupling ("your crate may compile on Modal but not locally") and offer both ladders explicitly: --manifest/--gpu… (S1, immediate) or --remote-describe (S2, automatic).
  3. S2 as the truth machine: remote describe also backs --verify-manifest (S7) so the silent F2/F3/F5 class is detectable even for users who never need the escape hatch.
  4. S6 in the docs now — costs nothing, prevents most F1 from existing.

Build order: S1 + the failure-message diagnosis are a small CLI change worth doing near-term; S2/S7 are one design (the bootstrap function + manifest cache) and slot naturally after the current parity work. S3/S4/S5 stay rejected/parked with reasoning recorded above.

6. Standing design rules this implies

  • Registration purity: #[function]/#[cls]/#[endpoint] registration must remain a pure function of source + resolved features — never of host env, build-script probes, or wall-clock (guards F5 by construction).
  • The manifest is a claim, not a fact: any consumer of a locally-built manifest must keep a degrade path for "the remote disagrees" (the input-validation degrade on unknown_entrypoint is the template).
  • Arch-independent data should have an arch-independent acquisition path available, even if the arch-dependent one stays the fast default — that is the whole page in one sentence.
