modal-rust — Architecture

Write an ordinary Rust fn, run / deploy / call it on Modal — through a first-party gRPC client. No modal CLI, no modal-rs, no per-project Python.

1. Vision & the one-paragraph mental model

modal-rust lets a user write a normal Rust function, annotate it (or register it by hand), and then run it three ways: in-process (.local()), remotely with a build-at-run-time (.remote()), or deployed once and called many times (deploy + call) — for CPU or GPU, trivial code or a real Burn/CUDA ML workload — all driven by our own lean Modal gRPC client.

The mental model. There are exactly two halves to the system, joined at one stable interface.

(1) The runner. A frozen contract: a user function is erased to a bare fn pointer in a Registry, and a tiny CLI binary (modal_runner) takes --entrypoint NAME --input-…, runs exactly one handler, and prints exactly one JSON envelope to stdout. This is the only thing that ever touches user code, and it never changes shape.

(2) The control plane. Everything that gets that runner onto Modal and invokes it: a first-party gRPC client (modal-rust-sdk) that authors a Modal FILE-mode function whose body is a tiny embedded Python wrapper. The wrapper builds the user's crate (in the function body for run, at image-build time for deploy) and execs modal_runner. Inputs/outputs cross as CBOR; the runner's JSON envelope rides back inside.

The seam between them — the runner CLI protocol — and the run-vs-deploy build boundary are the two invariants the whole project is organized to protect.

This document focuses on data flows and internal abstractions, and points at the real code. Every file:line below was read from the sources in this repo; the long-form milestone history lives in workpads/shim-backend/knowledge.md and the original frozen contract in workpads/architecture/boundaries.md (note: that contract doc predates the programmatic SDK; where it describes a generated-Python CLI it has been superseded — see §5).

2. Crate map — one responsibility each

A single virtual cargo workspace (Cargo.toml) with five library/binary crates and five examples. Edges are acyclic; the runtime crate sits at the bottom with zero Modal/network/Python dependencies.

modal-rust-runtime FROZEN

The runner contract. The Registry (BTreeMap<&str, HandlerFn>), typed!, RunnerError (5 kinds), run_cli + the file-mode runner CLI protocol, and optional dispatch-only inventory collection. Deps: only serde, serde_json, anyhow, inventory + a hand-rolled arg parser — no Modal, network, or Python, because it is recompiled on every run and baked into every deploy image.
crates/modal-rust-runtime/src/lib.rs (1113 lines)

modal-rust-macros

The #[modal_rust::function(...)] attribute. Purely additive sugar: it emits the unchanged user fn plus one facade-owned inventory::submit! of a Registration whose handler is the same typed!(f) wrapper the manual builder produces, paired atomically with the decorator's gpu/timeout/cache/secrets/volumes as static FunctionConfig that the facade converts to owned FunctionOptions.
crates/modal-rust-macros/src/lib.rs

modal-rust-sdk (modal_rust::sdk)

The lean first-party Modal gRPC client. Auth interceptor + TLS channel + CBOR codec + retry_transient, plus the typed ops/* surface (App, Image, Function authoring & invocation, Mount, local-dir upload + Blob, Volume, Secret). Vendors Modal's api.proto and compiles it with tonic-prost-build; depends on neither modal-rs nor the modal CLI.
crates/modal-rust-sdk/src/{client,auth,channel,codec,retry,error,config}.rs + ops/*

modal-rust (the facade)

The user-facing API. App / Function with .local() / .remote() / .spawn() / .map() / deploy / call; RemoteConfig / DeployConfig; the embedded run/deploy Python wrappers (as Rust strings); cargo-metadata upload scoping. Re-exports sdk, the runtime essentials, and the function macro.
crates/modal-rust/src/{lib,app,function,remote,deploy,scope,error}.rs

modal-rust-cli

The modal-rust binary. Cannot link the user crate, so it builds the user's modal_runner, reads modal_runner --describe to learn the entrypoints + per-fn config, then drives a headless App through the same facade ops. clap/tokio live here only.
crates/modal-rust-cli/src/{main,programmatic,doctor,workspace}.rs

examples/

add (manual registry), add-macro (the #[function] twin), cuda-vector-add (cudarc, dynamic-loading), burn-add (a real Burn/CubeCL GPU workload — CUDA-only, excluded from default-members), and orchestrate (a runnable tour of the facade).
examples/{add,add-macro,cuda-vector-add,burn-add,orchestrate}

The workspace pins panic = "unwind" for both dev and release profiles (Cargo.toml), because the runner upgrades panics into structured envelopes via catch_unwind — panic = "abort" would silently degrade that to a raw process abort.

3. Core abstractions

3.1 The Registry, HandlerFn & typed! — static dispatch, no dyn

A handler is erased to a bare function pointer. There is no Box<dyn Handler> and no vtable anywhere; the registry is the project's one "open set", and it is closed over fn pointers.

// crates/modal-rust-runtime/src/lib.rs:33
pub type HandlerFn = fn(&[u8]) -> Result<Vec<u8>, RunnerError>;

// :340  — keys are &'static str, values are fn pointers: no alloc, no dyn
pub struct Registry { handlers: BTreeMap<&'static str, HandlerFn> }

typed!(f) is a macro_rules! that generates a monomorphized wrapper fn for the concrete In/Out/Err of f and yields its pointer. Decode → call → encode are all inlined per handler (crates/modal-rust-runtime/src/lib.rs:189):

macro_rules! typed {
  ($f:path) => {{
    fn __wrap(input: &[u8]) -> Result<Vec<u8>, $crate::RunnerError> {
      let arg = $crate::codec::decode(input)?;        // In inferred from $f
      match $f(arg) {
        Ok(out) => $crate::codec::encode(&out),         // Out inferred from $f
        Err(e)  => /* wraps the user error onto RunnerError::Function */
      }
    }
    __wrap as $crate::HandlerFn
  }};
}

The two registration paths converge on this same wrapper. The manual builder (examples/add/src/lib.rs:152):

Registry::new().function("add", typed!(add))   // duplicate names panic at startup

…and the macro path, which submits the identical pointer via inventory and is collected by Registry::from_inventory() (crates/modal-rust-runtime/src/lib.rs:366). The codec is a JSON Codec over bytes (:161); a future CBOR wire format would add a sibling impl and never touch HandlerFn or Registry.

Why it matters. Because the handler is a fn pointer and the codec is byte-neutral, the manual builder, the proc-macro, async (reserved typed_async!), and alternate wire formats are all additive — none of them can change the dispatch shape run_cli depends on.

3.2 RunnerError & the failure envelope — frozen at five kinds

The runner models failure as a Rust enum that wraps the user error rather than stringifying it early (crates/modal-rust-runtime/src/lib.rs:39):

kindcausevariant
decode_errorinput not valid JSON, or valid JSON that won't deserialize into InDecode(String)
unknown_entrypoint--entrypoint name absent from the registryUnknownEntrypoint(String)
function_errorhandler returned Err(_) — the user error wrappedFunction { message, details }
encode_errorOut failed to serialize (e.g. non-string map keys) — must not be a panicEncode(String)
panichandler unwound; message + backtrace capturedPanic { message, backtrace }

The function_error kind preserves structure: when the handler's error type is Serialize, details = serde_json::to_value(&e); for opaque errors like anyhow::Error, details is null and only the Display/anyhow chain rides in message. The typed! macro picks the right path at compile time via inherent-vs-trait method specialization (Wrap<E> in __macro_support, :221).

The success/failure envelopes are emitted by run_cli_dispatch (:646), one JSON line to stdout, exit code mirroring ok:

{"ok":true, "value":<json>}
{"ok":false,"error":{"kind":"…","message":"…","details":<json|null>,"backtrace":"…"}}

Precedence is frozen: top-level JSON parse → entrypoint lookup → decode In → call → encode Out (:665–704). A malformed-JSON input is a decode_error even when the entrypoint is also unknown. Panic capture uses a process-wide hook installed once via Once, writing into a thread_local! slot (:421–491) with Backtrace::force_capture(), so parallel panics never race and the backtrace is populated regardless of RUST_BACKTRACE.

The facade's parse_envelope (crates/modal-rust/src/remote.rs:657) reconstructs this exact enum on the client side, so a remote failure is identical in shape to a local one — an unknown kind degrades to Decode, never a panic.

3.3 FunctionConfig — the decorator is the config

Per-function deploy/run config lives on the facade-owned Registration alongside the handler (crates/modal-rust/src/registration.rs), while the runtime registration remains dispatch-only:

pub struct FunctionConfig {
  pub gpu:          Option<&'static str>,                  // "T4", "A100-80GB", "H100:4"
  pub timeout_secs: Option<u32>,
  pub cache:        Option<bool>,
  pub secrets:      &'static [&'static str],            // named Modal secrets
  pub volumes:      &'static [(&'static str, &'static str)], // (mount_path, name)
}
pub struct FunctionOptions {
  pub gpu:          Option<String>,
  pub timeout_secs: Option<u32>,
  pub cache:        Option<bool>,
  pub secrets:      Vec<String>,
  pub volumes:      Vec<(String, String)>,
}

This is facade metadata: the runner crate never defines or reads these fields. FunctionConfig is only the const-friendly inventory initializer. Immediately after inventory collection, the facade converts it into owned FunctionOptions; App, RemoteConfig.options, DeployConfig.options, the deploy plan, dry-run, and the CLI manifest all carry that one owned type. The fields are &'static in FunctionConfig because inventory::submit! builds a static initializer that must be const-constructible. A bare #[modal_rust::function] yields default options, so it is wire-identical to before these fields existed.

The macro parses the attribute and emits the config straight into the registration (crates/modal-rust-macros/src/lib.rs:246). See it used live in examples/add-macro/src/lib.rs:62: #[modal_rust::function(gpu = "T4", timeout = 1800, cache = false)].

3.4 The SDK client, the ops/* surface, the 3 fixes & retry_transient

ModalClient (crates/modal-rust-sdk/src/client.rs:39) wraps the tonic-generated stub behind an AuthInterceptor over a hardened TLS channel. connect() reads ~/.modal.toml / MODAL_TOKEN_*, builds the channel, and fires a ClientHello handshake to fail fast on bad credentials. Auth is by request metadata, not a login RPC: every call carries x-modal-token-id/secret, x-modal-client-type = "1", version, platform, timestamp (crates/modal-rust-sdk/src/auth.rs:56).

The control-plane operations are a native port of the proven file-mode recipe, each its own submodule under ops/, all impl ModalClient blocks so the public surface is method calls on one client (crates/modal-rust-sdk/src/ops/mod.rs):

moduleRPCs
ops::appAppGetOrCreate / AppCreate (ephemeral) / AppPublish
ops::imageImageGetOrCreate + ImageJoinStreaming build poll
ops::mountMountGetOrCreate (GLOBAL) → client mount & python-standalone mount
ops::local_dir / ops::blobMountGetOrCreate + MountPutFile (inline) / BlobCreate + HTTP PUT (≥4 MiB)
ops::functionFunctionPrecreate + FunctionCreate (FILE) + FunctionGet
ops::invokeFunctionMap → FunctionPutInputs fallback → FunctionGetOutputs
ops::volume / ops::secretVolumeGetOrCreate (V2) / SecretGetOrCreate

Three bugs in the reference modal-rs client were diagnosed against Modal's live server and baked into this SDK as the three fixes:

  1. Fix #1 — FunctionCreate XOR. Modal expects exactly one of function / function_data and always wants resources set. The SDK sends the single-Function path with resources always populated, function_data = None, and existing_function_id = precreate_id to legalize an empty function_serialized in FILE mode (crates/modal-rust-sdk/src/ops/function.rs:326).
  2. Fix #2 — deploy via AppPublish only. The legacy AppSetObjects RPC handler is server-broken; modern deploys publish with AppPublish alone (crates/modal-rust-sdk/src/ops/app.rs:114).
  3. Fix #3 — FunctionPutInputs fallback. If FunctionMap's response does not echo the pipelined input, it was never enqueued; the SDK then calls FunctionPutInputs before polling outputs (crates/modal-rust-sdk/src/ops/invoke.rs:188).

Every unary RPC (and every per-file upload) is wrapped in retry_transient (crates/modal-rust-sdk/src/retry.rs:68): 100 ms→5 s exponential backoff with full jitter, 8 attempts, a 120 s cap — mirroring Modal's own retry_transient_errors. Crucially, it retries only transient errors (Error::is_transient: connection reset, h2 protocol error, UNAVAILABLE/Internal/Unknown); real errors (auth, invalid-argument, in-band build/function failure) propagate on the first occurrence and are never masked.

3.5 ImageSpec & FunctionSpec — declarative builders

ImageSpec (crates/modal-rust-sdk/src/ops/image.rs:74) is a declarative recipe that renders a Dockerfile-command list. Its dockerfile_commands() (:289) emits the official client's add_python branch — COPY /python/. /usr/local, an auto ln -s python3 python for series < 3.13, and the TERMINFO_DIRS ENV — then optionally a rustup+CUDA block (with_rust_toolchain), then base64-baked wrapper modules, then any extra commands. Builders: from_registry, with_add_python, with_python_standalone_mount_id, with_wrapper_module, with_command, with_context_mount, with_base_image (layered FROM base).

FunctionSpec (crates/modal-rust-sdk/src/ops/function.rs:126) describes a FILE-mode function: module_name + function_name (no serialized bytecode), image_id, mount_ids, timeout, resources (GPU via with_gpu → parse_gpu_config, mirroring Modal's _utils/function_utils.py), mount_client_dependencies = true (worker injects the modal client's pip closure at boot), plus volume mounts and secret ids. Empty volume/secret lists make prost omit those fields, keeping the wire bytes identical for functions that use neither.

3.6 The App / Function facade & RemoteConfig / DeployConfig

App (crates/modal-rust/src/app.rs:26) owns a Registry, a per-name configs map, and an optional live RemoteHandle (a Mutex<ModalClient> + ephemeral app_id + a function_ids map — one OnceCell<function_id> per entrypoint, keyed by the entrypoint and its effective config, each single-flighting its own create). Each entrypoint becomes its own Modal function (distinct object tag = the entrypoint name, decoupled from the in-container handler callable), so functions with different gpu/timeout/cache/secrets/volumes coexist without overriding one another. Constructors form a clear matrix:

A Function<'a> (crates/modal-rust/src/function.rs:17) is a cheap handle: app.function("add") resolves the HandlerFn from the registry now (Option, so an unknown name surfaces a clear error only when an op is actually invoked) and keeps the name. 4. The data flows

Each flow below is a numbered step-through with the exact functions and files. The first two are entirely local; the rest cross to Modal.

4.1 Authoring → the inventory Registry

Two routes, one registry shape.

  1. Manual: the user writes pub fn modal_registry() -> Registry and the runner bin calls run_cli(modal_registry()) (examples/add/src/bin/modal_runner.rs:9). Each .function(name, typed!(f)) rejects a duplicate name with a hard startup panic.
  2. Macro: #[modal_rust::function] expands to the unchanged fn plus one facade-owned inventory::submit!{ Registration { name, handler: typed!(f), config, package } } (crates/modal-rust-macros/src/lib.rs). The runner bin calls modal_rust::run_cli_from_inventory().
  3. Convergence: modal_rust::registry_from_inventory() inserts each facade submission through the same Registry::function path, producing a map byte-identical to the hand-built one. examples/add-macro's tests assert the success envelope is byte-for-byte the manual path's.

4.2 .local() — in-process dispatch zero Modal

The cheapest, highest-value path: run the registered handler in-process exactly as the runner would, minus the subprocess (crates/modal-rust/src/function.rs:42):

pub fn local<In: Serialize, Out: DeserializeOwned>(&self, input: In) -> Result<Out> {
  let handler = self.handler.ok_or_else(|| self.unknown())?;
  let bytes = serde_json::to_vec(&input).map_err(Error::Encode)?;
  let out = handler(&bytes).map_err(Error::Runner)?;   // the SAME HandlerFn the runner calls
  serde_json::from_slice(&out).map_err(Error::Decode)
}

Input → JSON → the handler's own codec::decode → In; the handler's codec::encode → JSON → Out. The double round-trip is intentional: it is identical to running the runner with no process boundary and no network. Errors map onto the same five-kind taxonomy. App::new(modal_registry()).function("add").local(AddInput{40,2}) yields AddOutput{sum:42} — see examples/orchestrate/src/main.rs:44.

4.3 .remote() — the run path, end to end run

This is the heart of the system: the user's crate is uploaded, cargo build runs in the function body at invoke time, and the freshly built modal_runner execs the handler. The result returns typed, with the exact semantics of .local().

  1. Connect. App::connect(name) builds a ModalClient (reads ~/.modal.toml / MODAL_TOKEN_*, ClientHello) and creates an ephemeral app via app_create_ephemeral (crates/modal-rust/src/app.rs:163). Ephemeral matters: the app is GC'd on client disconnect, so a run never leaves a lingering persistent deploy.
  2. Resolve config & ensure the function (once per entrypoint). Function::remote serializes the input and calls App::remote_invoke → resolve_function (app.rs:361), which reads this entrypoint's owned FunctionOptions, folds cache/timeout defaults once, stores the effective options on the run config, then runs ensure_function once for this entrypoint via its own function_id OnceCell (single-flighted under concurrent calls). Each entrypoint is created as its own Modal function — object tag = the entrypoint name, the in-container callable stays handler — so divergent per-entrypoint configs never override one another.
  3. Scope the upload. scope::workspace_closure (crates/modal-rust/src/scope.rs:91) shells cargo metadata --no-deps at the workspace root and DFS-walks the target package's workspace-member normal path-dep closure — exactly the crate dirs cargo build -p <pkg> needs — plus the workspace Cargo.lock and a rewritten Cargo.toml (members trimmed to the closure, format-preserving via toml_edit, panic="unwind" kept). Everything else (sibling crates, vendored clones, docs) is excluded by construction. On any failure it returns None → whole-root fallback.
  4. Upload as a mount. mount_workspace_closure (crates/modal-rust-sdk/src/ops/local_dir.rs:115) walks the chosen dirs, prunes by ignore precedence .modalignore > .gitignore > built-in defaults (a real gitignore engine), sha256s each surviving file, and uploads it: inline via MountPutFile for small files, or BlobCreate + an HTTP PUT to object storage for files ≥ 4 MiB (ops/blob.rs:39). One MountGetOrCreate finalizes the ephemeral mount → mount_id. Each file lands at /src/<rel>.
  5. Resolve the platform mounts. client_mount_id resolves the hosted, version-keyed modal-client-mount-{version} (GLOBAL namespace, pure lookup) so modal is importable in the container; python_standalone_mount_id resolves python-build-standalone.{release}.{full}-gnu for add_python (crates/modal-rust-sdk/src/ops/mount.rs).
  6. Build the run image. image_get_or_create with an ImageSpec: from_registry("rust:1-slim") + with_add_python("3.12") + the standalone mount as build context + the base64-baked run wrapper + ENTRYPOINT [] (crates/modal-rust/src/remote.rs:565–588). The image carries no cargo line — that is the run-side build boundary. The build streams via ImageJoinStreaming; the builder version is resolved from the environment (must be > "2024.10" so the worker mounts the client's pip closure at boot).
  7. Create & publish (FILE mode). function_precreate → function_create with definition_type = FILE, empty function_serialized, module_name = modal_rust_run_wrapper, function_name = handler, mount_ids = [client, source], resources.gpu_config from the decorator, then app_publish_ephemeral (remote.rs:591–643). Publishing is required to make the function invokable; the ephemeral state keeps the app throwaway. The created function_id is invoked directly — an ephemeral app is not name-resolvable.
  8. Invoke. invoke_cbor_with_deadline encodes (args, kwargs) where args = (entrypoint, input_json) as CBOR and drives FunctionMap → (fix #3) FunctionPutInputs → poll FunctionGetOutputs (crates/modal-rust-sdk/src/ops/invoke.rs). The deadline is the function's container timeout + 120 s, because the first call pays a cold in-body cargo build that can run for minutes (app.rs:284).
  9. Decode. The wrapper returns the runner's one-line JSON envelope as a string; it rides back as CBOR; Function::remote hands it to parse_envelope::<Out> (remote.rs:657), which decodes value into Out on success or reconstructs the frozen RunnerError on failure — identical to .local().

Live evidence (knowledge.md): add(40,2).remote() = AddOutput { sum: 42 } — Modal built the crate in the function body (Compiling example-add … Finished release in 14.37s, modal_runner exit=0) and returned the real Rust result, with no modal CLI, no per-project .py, no modal-rs.

4.4 The embedded Python wrapper, the client mount & container boot

Why is there Python at all? Because a Modal FILE-mode function is identified by module_name + function_name, and the container boots through python -m modal._container_entrypoint, which does importlib.import_module(module_name) + getattr(module, function_name) and calls it. So something importable must be on the image. We ship one tiny wrapper module (baked into the image, not per-project) and let the client mount make modal importable.

  • The run wrapper is a Rust const WRAPPER_SRC with a {{PACKAGE}}/{{CACHE}} placeholder, baked base64 into the image (no shell quoting). Its handler(entrypoint, input_json) picks a writable build dir, runs cargo build --release -p PKG --bin modal_runner in the function body, writes /tmp/in.json, execs modal_runner --entrypoint … --input-file /tmp/in.json, and returns the stdout envelope verbatim (crates/modal-rust/src/remote.rs:60–225). All cargo/rustc/user diagnostics go to stderr — load-bearing, since stdout must carry exactly one JSON line.
  • The client mount is the Modal-native way to make modal importable: a hosted, version-keyed GLOBAL mount of the modal source at /pkg, attached via Function.mount_ids (crates/modal-rust-sdk/src/ops/mount.rs) — exactly what the official Python SDK attaches automatically. The mount carries only the source; the modal client's third-party pip closure (typing_extensions, grpclib, …) is injected by the worker at container start when mount_client_dependencies = true (proto field 82, only honored for builder version > "2024.10"). So there is no pip install modal on the modern path.
  • Python itself comes from the hosted python-standalone mount via add_python — relocatable and not PEP-668-managed, dissolving the early python-is-python3 / --break-system-packages hacks. The apt+pip path survives only as a documented fallback selected when add_python is unset (crates/modal-rust-sdk/src/ops/image.rs:289–337).

4.5 deploy + call — build at image-build time deploy

Deploy proves the other half of the build boundary. The source is COPYed into an image layer and cargo build runs during the image build; the deployed runtime then execs only the prebuilt /app/modal_runner and never invokes cargo.

  1. Upload & layer. deploy_function (crates/modal-rust/src/deploy.rs:267) uploads the source the same way (cargo-scoped) but as the image build context landing at /app/src. It builds two layers: deploy_base_layer_spec (add_python) then deploy_top_layer_spec (FROM base, bake the deploy wrapper, COPY . /, RUN cargo build --release -p PKG --bin modal_runner, cp to /app/modal_runner).
  2. Create & publish persistently. A FILE-mode function with the client mount only (no source mount — the binary is baked) and app_publish_deployed into a stable, get-or-created name (re-deploys replace in place). Returns a DeployedApp.
  3. Call. call_function (deploy.rs:403) does function_from_name + invoke_cbor_with_deadline — no upload, no image build, no publish. The deploy wrapper (deploy.rs:65) imports only subprocess/sys, has no cargo, no /src, no CARGO_*, and execs only /app/modal_runner. A unit test asserts that runtime purity.
Proof obligation, met. cargo build appears in the deploy/build logs and is absent from call logs. Live: the base + top layers built at image-build time (Finished release in 9.07s → binary baked to /app/modal_runner) and the call phase ran zero cargo. The capstone repeated this with a real Burn/CubeCL workload on a Tesla T4.

4.6 .spawn() / .map() — fire-and-forget & fan-out run

Both reuse the same ensure-function head as .remote(), then diverge at the invoke layer (crates/modal-rust-sdk/src/ops/invoke.rs):

  • spawn sends a UNARY FunctionMap with FunctionCallInvocationType::Async and returns the function_call_id immediately (spawn_cbor, :325). FunctionCall::get later polls that call's output at index 0 (get_by_call_cbor) and decodes through the same parse_envelope.
  • map opens a FUNCTION_CALL_TYPE_MAP call, enqueues N inputs each tagged with its ordinal as idx, polls outputs, and reassembles them in input order by idx into a BTreeMap (map_cbor, :453). Fail-fast: the first remote failure surfaces immediately.

Both poll FunctionGetOutputs seeding the cursor with the "0-0" sentinel that Modal's server requires on an indexed get (LAST_ENTRY_ID_INITIAL, :45). Live: map([{1,1},{2,2},{3,3},{40,2}]) → [2,4,6,42] in order.

4.7 The build cache — one archive on a V2 volume run

The run-path in-body cargo build is cached, on by default. Modal volumes degrade past ~50k files, so the cache is not CARGO_HOME/target directly on a volume; instead it is one compressed archive (cache.tar.zst, gzip fallback) on a V2 volume at /cache (crates/modal-rust/src/remote.rs:104–164):

  1. On container start, the wrapper unpacks the archive to /tmp (build on fast local disk: CARGO_HOME=/tmp/cargo, CARGO_TARGET_DIR=/tmp/target).
  2. After the first cold build it repacks the changed dirs into one object (atomic temp+replace), relying on allow_background_commits — never vol.reload() on the hot path.
  3. The decorator #[function(cache=false)] overrides the env/default base; MODAL_RUST_NO_CACHE opts out. A cache miss only costs time, never correctness.

The volume is resolved via volume_get_or_create (V2) and attached at /cache in ensure_function (remote.rs:458–471, 611–613) — only on the run path; deploy attaches no cache volume. Live: cold cargo build 6.45s → warm 0.06s (Fresh, no recompile), ~3.5× end-to-end on add (the real win lands on heavy builds).

4.8 Secrets & user volumes

#[function(secrets = ["name"], volumes = ["/mount=name"])] rides the FunctionConfig into both run and deploy. In ensure_function / deploy_function, each secret name is resolved via secret_get_or_create → Function.secret_ids (Modal injects the secret's key/values as container ENV VARS), and each user volume via volume_get_or_create (V1) → a FunctionVolumeMount at its mount path — a separate mount from the /cache cargo cache, with a guard rejecting a user mount at /cache (crates/modal-rust/src/remote.rs:473–501, 616–623). Live: a fn read std::env::var("MODAL_RUST_TEST_SECRET") and wrote a marker to /data on a user volume that a fresh container read back — real cross-call persistence.

4.9 The CLI programmatic path

The modal-rust binary cannot link the user crate, so it learns the entrypoints + per-fn config out-of-band via --describe (crates/modal-rust-cli/src/programmatic.rs):

  1. Build for the manifest. cargo build --release -p <pkg> --bin modal_runner at the workspace root (the local binary is for the manifest only and is never uploaded — the remote build still happens per the boundary).
  2. Describe. Run modal_runner --describe and parse the modal-rust/describe@1 manifest: each entrypoint's name + serialized facade FunctionConfig (crates/modal-rust/src/registration.rs). --describe is an additive facade-runner subcommand; when absent, the frozen runtime --entrypoint dispatch is byte-identical (the first token differs, so they never collide).
  3. Drive a headless App. Build App::from_manifest / connect_from_manifest (config but no handlers) and call the same facade ops the facade uses: run → remote_envelope, deploy → deploy_with, call → call_envelope (crates/modal-rust/src/app.rs:362–379). The envelope string is printed verbatim and ok mirrors the exit code.

The CLI emits no generated .py and spawns no modal subprocess — the only subprocesses are cargo and the user's modal_runner. doctor checks credentials (and, with --rust, cargo/rustc + panic = "abort" detection).

5. The frozen seams & why

Two invariants are guarded across every change. Everything else is built to be additive around them.

Seam A — the runner CLI protocol

modal_runner --entrypoint NAME (--input-json | --input-file | --input-stdin) → exactly one JSON envelope on stdout, diagnostics on stderr, exit code mirroring ok, five frozen error kinds. The handler shape fn(&[u8]) -> Result<Vec<u8>, RunnerError>, typed!, and Registry never change.

  • Why one JSON line on stdout: the Python wrapper parses the result robustly despite arbitrary cargo/rustc/user build noise (which all goes to stderr).
  • Why exactly five kinds, with encode_error distinct from panic: an output-serialization failure must not masquerade as a crash; the taxonomy is the stable contract the client reconstructs (parse_envelope).
  • Why fn pointers, not dyn: the manual builder, the proc-macro, async, and CBOR are all additive precisely because dispatch is one monomorphized indirect jump that cannot change shape.
  • Additive proof: --describe was added without touching the dispatch path — it is a strict superset gated on the first argv token (crates/modal-rust-runtime/src/lib.rs:631), and examples/add-macro asserts byte-identical envelopes across manual/macro.

Doc-vintage note. workpads/architecture/boundaries.md is the research-stage contract: §2/§3 (the runner protocol, error taxonomy, registry/typed! shapes) and §4 (the build boundary) remain authoritative and exactly describe the code. But its §5/§8 describe a since-superseded design where the CLI generated Python shims and shelled out to the official modal CLI. That entire control layer was replaced by the first-party modal-rust-sdk + the programmatic facade (the pivot is narrated in workpads/shim-backend/knowledge.md). The frozen seams above survived the pivot unchanged — that was the whole point.

6. How to contribute — where things live

The codebase is organized so that the three most common kinds of change each touch a predictable, small set of files. Run the offline gates before and after: cargo fmt --check, cargo clippy --all-targets --all-features -- -D warnings, cargo test (live tests are #[ignore]d behind a live feature, so CI never hits Modal).

Add a new Modal RPC

  1. Find the message/method in the vendored crates/modal-rust-sdk/proto/api.proto (compiled by build.rs into proto::api). If it isn't there, sync it from references/modal-client/modal_proto/api.proto.
  2. Add an async fn on impl ModalClient in the right ops/*.rs module, wrapping the call in retry_unary(name, …) (clone the stub per attempt). Cross-check the request shape against the official client in references/modal-client/py/modal/ — that is how the three fixes were found.
  3. Re-export any public types from crates/modal-rust-sdk/src/lib.rs, and add unit tests (the ops modules test request-building offline).

Add a new decorator argument

  1. Add the field to facade FunctionConfig (crates/modal-rust/src/registration.rs) as a const-constructible type (an Option<…> or &'static slice), and add the owned equivalent to FunctionOptions with any needed #[serde(default)] for old-manifest compat. Do not add Modal vocabulary to modal-rust-runtime.
  2. Parse it in crates/modal-rust-macros/src/lib.rs (the syn::meta::parser block) and emit it into the inventory::submit!.
  3. Read it from FunctionOptions in the facade: fold it in resolve_function and the per-entrypoint deploy plan (crates/modal-rust/src/app.rs), then apply it per entrypoint in ensure_function/deploy_function via the relevant FunctionSpec/ImageSpec builder.
  4. The CLI picks it up by deserializing the --describe manifest into FunctionOptions. Keep the bare-decorator path byte-identical (assert it, like bare_macro_config_is_default).

Add a new image step

  1. Add a builder on ImageSpec (a field + a with_* method) and render it in dockerfile_commands() (crates/modal-rust-sdk/src/ops/image.rs:289), respecting the existing ordering (add_python branch → rust toolchain → wrapper bakes → extras) so the default path stays byte-identical.
  2. Wire it from the facade where the image is assembled: ensure_function for the run image (crates/modal-rust/src/remote.rs:565) and/or deploy_{base,top}_layer_spec for deploy (crates/modal-rust/src/deploy.rs:213,239).
  3. Assert the rendered commands SDK-side (dockerfile_commands is private to the SDK) and the public ImageSpec fields facade-side.
The golden rule. Anything that would change the runner CLI protocol, the five error kinds, the HandlerFn/Registry/typed! shapes, or the run-vs-deploy boundary is not a normal change — it touches a frozen seam and must be reviewed against §5. Almost everything worth adding can be done additively around them.
