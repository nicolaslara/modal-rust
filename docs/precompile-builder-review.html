Architecture Proposal Affects RUN Affects DEPLOY

Precompile on a Modal Builder vs. Inline Compile

Decoupling the Rust build into a dedicated, reusable Modal container that emits a fingerprinted binary — which run and deploy then fetch and exec.

modal-rust · architecture review · grounded in workpads/architecture/boundaries.md §2/§4/§5/§7 and the implemented control_plane.rs Boundary / SourceDelivery model · decision owner: project lead

TL;DR for the reviewer
The proposal does not change whether we compile on Modal (we already do). It changes where: it lifts the build out of the run-function-body and the deploy-image-layer into a separate, long-lived builder container, producing a binary keyed by a content fingerprint. Both run and deploy become "fetch the prebuilt binary, exec it." Both hard design stances survive. The win is concentrated on the GPU path (compile on cheap CPU, not GPU minutes), the run dev-loop (the P7 cache becomes a first-class warm builder), and deploy image size/rebuild-cascade. The new costs are cross-arch correctness, artifact identity discipline, and a second cold-path hop. Recommendation: spike it, scoped to GPU + run first — see §9.

1 · The decision being asked

Today, the Rust compile is coupled to the operation that needs it. The proposal is to make compilation its own phase whose only output is a binary, addressable by fingerprint, that any later operation reuses.

The original idea, verbatim in spirit: "compile on Modal as a separate step in the same type of machine that we will run, then send the binary on run/deploy, and reuse the compilation container for future compilation." The reviewer is being asked to decide whether this is worth a research spike, and if so, scoped to what. This document compares it head-to-head with the implemented design so that decision can be made on mechanics, not vibes.

3 · The proposed builder model

Introduce a third phase, BUILD, owned by a dedicated Modal Function (or Sandbox) that lives across invocations. It holds a warm target/ on local SSD (fast incremental) backed by a Volume snapshot for durability across container recycles. It is the single writer of the artifact store, keyed by the existing P7 fingerprint:

fingerprint = H(source after ignore rules
              + Cargo.lock + toolchain/image + profile
              + target-triple + features + runner/generator version)

RUN and DEPLOY stop compiling. They resolve the fingerprint, ask the builder for (or wait on) the matching binary, and consume it:

BUILDER (warm, single writer) consumers ────────────────────────── ───────── fingerprint?──hit──→ return path RUN: fetch binary from Volume → exec (mutable, fresh per change) └─miss─→ incremental DEPLOY: BAKE that exact binary into a thin cargo build → COPY-only image layer → exec (immutable, no Volume at call time) publish binary keyed by fp

The run-vs-deploy distinction collapses from "body-cargo vs image-cargo" to a single, cleaner axis — artifact durability:

RUN pulls a mutable, fresh fingerprinted binary from the builder Volume. Always reflects current source.
DEPLOY bakes the exact fingerprinted binary into an immutable image layer. Self-contained; no call-time Volume dependency.

5 · Do the frozen invariants survive?

This is the crux for the reviewer. The proposal is an implementation refactor of "where build runs," not a contract change.

Invariant (source)Survives?Why
Deployed runtime never invokes cargo (§2, §4) YES Deploy still execs only a prebuilt binary. Build moves from the image RUN layer to the builder; the call path is untouched.
run builds late / deploy bakes immutable (§2 stance 2) YES — sharper Run gets a fresh fingerprinted binary per source change; deploy bakes an immutable one. The boundary becomes a clean durability axis.
Direct-execution-first; Sandbox is fallback (§stance 1) YES The builder is a normal @app.function. Only escalate to a Sandbox if a Function body genuinely can't build (same rule as today).
Frozen runner CLI seam (§2) + registry/typed! (§3) YES These govern what the binary does, not where it's built. Zero changes.
M8 proof: cargo in build logs, absent from call logs RELOCATES cargo now appears in builder logs, not image-build logs. Spirit preserved (no cargo at call time); the acceptance assertion must be re-pointed at the builder. Doc change, not a violation.
P7 "cache is advisory; hit only on exact fingerprint" (§7) PROMOTED to primary Today a bad cache hit costs only time. In the builder model a wrong fingerprint means you exec the wrong binary. The exact-match rule becomes a load-bearing correctness invariant, not a side optimization.
Reviewer takeaway
Nothing in the frozen contract forbids this. The two items that move — M8's log location and P7's cache rule — are exactly the two places to focus review effort, because they shift from "convenience" to "correctness."

6 · Latency & cost analysis

Run dev-loop

Warm hit: faster — skip cargo entirely, fetch + exec.

Cold/changed: ~same or slightly worse if builder≠runner (extra fetch hop), but the builder's warm local target/ beats P6's volume-backed incremental.

GPU path

Big win. Per §9, the binary is mostly CPU code that dlopens CUDA at runtime (cudarc dynamic-loading). Tier 0/1 binaries need no GPU to compile. Compile on a cheap CPU builder; spend GPU minutes only on the actual run. Only Tier 2 (nvcc) needs CUDA headers — still no GPU.

Deploy

Win. Image becomes a thin COPY binary layer instead of toolchain + source + cargo. Smaller, faster to build, and the §12 cascading-rebuild risk largely evaporates — incremental rebuilds live in the builder, not in opaque image layers.

7 · New failure modes the reviewer must weigh

Cross-arch SIGILL
The one safety inline-build gives for free is build-CPU == run-CPU. A separate builder must pin arch/CPU-feature parity (and avoid -C target-cpu=native unless builder==runner class), or ship a binary that SIGILLs at exec. Mitigation: a startup self-check that verifies the binary's target triple/arch vs the host before exec — analogous to the §9 CUDA dlopen self-check.
Fingerprint is load-bearing
Reuse is correct only on an exact match of the full P7 input set. A stale binary = wrong code, not a slow rebuild. Mitigation: conservative fingerprint (include toolchain, profile, features, triple, runner/generator version), and make a fingerprint miss always cheap-and-correct.
Extra cold-path hop
If builder≠runner, a cold first run pays builder cold-start + build + artifact sync + runner fetch. See the fork in §8 — the answer is to default builder==runner on the CPU run path and only split for GPU/deploy.
"Warm container" is best-effort
Modal containers are ephemeral; local target/ is not guaranteed across recycle. "Warm builder" realistically = warm container (min_containers≥1 / long scaledown) for speed while alive + a Volume snapshot for durability. Don't assume warm disk persists.
Deploy immutability tension
A deployed app must not depend on a mutable Volume at call time (breaks self-containment + reload semantics). The builder Volume is the source of the bake, never a call-time dependency. Deploy must COPY the binary into the image at deploy time.
Artifact store surface
A new component (Volume layout, GC, fingerprint index, single-writer lock) to design, test, and operate. Real engineering cost that the current "image layers are the cache" model avoids entirely.

9 · Recommendation & spike plan

Recommendation
Approve a scoped research spike, not a wholesale rewrite. Adopt builder == runner on the CPU run path first (it's P7 made real — lowest risk, immediate dev-loop win), and prototype the separate CPU builder specifically for the GPU + deploy paths, where the hop cost is dwarfed by the GPU-minute and image-size savings. Keep the inline path as the fallback until the spike clears the three gates below.

Phased adoption

Phase A — Run, builder==runner
Formalize "build once, keep warm, skip on fingerprint hit." Smallest delta from today. Validates the fingerprint + artifact contract with zero cross-arch risk.
Phase B — GPU, builder≠runner
Cheap CPU builder emits the GPU-run binary. This is where the model earns its keep. Requires the cross-arch self-check.
Phase C — Deploy bakes builder output
Thin COPY-only deploy image sourced from the builder Volume. Re-point the M8 proof at builder logs.

Gates the spike must clear before ratifying

1Warm-disk reality: does a warm Modal container actually retain target/ long enough to matter, or is the Volume snapshot the real mechanism? Measure incremental-rebuild latency warm vs cold-recycle.

2Cross-arch parity: what does Modal actually schedule the builder on, and can we pin it to the run target's arch/CPU class? Prove the startup self-check catches a deliberate mismatch instead of SIGILL.

3Fingerprint round-trip correctness: a non-trivial source change must miss the cache and produce a correct new binary; an identical re-run must hit. No false hits under feature/profile/toolchain perturbation.

Where this slots in
There is already a precompile-research skill and a runner-approach-research lineage. This belongs as a new architecture workpad alongside workpads/architecture/boundaries.md, gated empirically the same way runtime-compile and mount-writability were (the project's established method).
